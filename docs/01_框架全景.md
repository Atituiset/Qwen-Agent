
Qwen-Agent 在生态里的定位：

| 维度      | LangChain        | AutoGen          | **Qwen-Agent**                  |
|-----------|------------------|------------------|---------------------------------|
| 核心理念  | 通用编排链        | 多 Agent 对话     | **模型原生能力优先**            |
| Tool 接入 | 自定义接口        | 自定义接口        | **FnCall + MCP 双通道**          |
| 模型绑定  | 模型无关          | 模型无关          | **深度适配 Qwen 系列**          |
| RAG 能力 | 外接向量库        | 外接向量库        | **内置文档解析+混合检索**        |
| 生产验证 | 社区驱动          | 学术驱动          | **Qwen Chat 后端**               |

**核心设计哲学** 一句话：不过度抽象，让 LLM 的原生 Function Call 能力做主力，框架做胶水和编排。后面所有"为什么不那样"都从这条推出来。

## §2. Agent 基类：`run` 与 `_run` 的两层契约

**决策点**：`Agent` 是抽象基类，**`run()` 永远不子类重写**（做消息类型归一、system message 注入、yield 协议转换），**`_run()` 永远子类必须实现**（具体工作流）。

**代码证据**（`qwen_agent/agent.py`）：
```python
# agent.py:31-44
class Agent(ABC):
    """A base class for Agent.
    An agent can receive messages and provide response by LLM or Tools.
    Different agents have distinct workflows ... in the `_run` method.
    """
    def __init__(self, function_list=None, llm=None, system_message=DEFAULT_SYSTEM_MESSAGE,
                 name=None, description=None, **kwargs):
        if isinstance(llm, dict):
            self.llm = get_chat_model(llm)        # 字典 → 模型实例
        else:
            self.llm = llm
        ...
        if function_list:
            for tool in function_list:
                self._init_tool(tool)             # 工具初始化三分支（见 §3）

# agent.py:78-131  run() 的实现
def run(self, messages, **kwargs):
    messages = copy.deepcopy(messages)            # ★ deepcopy 防污染
    _return_message_type = 'dict'
    new_messages = []
    if not messages:
        _return_message_type = 'message'
    for msg in messages:
        if isinstance(msg, dict):
            new_messages.append(Message(**msg))
        else:
            new_messages.append(msg)
            _return_message_type = 'message'      # ★ 只要有 Message 就改回 message

    if 'lang' not in kwargs:
        if has_chinese_messages(new_messages):
            kwargs['lang'] = 'zh'
        else:
            kwargs['lang'] = 'en'

    if self.system_message:
        if not new_messages or new_messages[0][ROLE] != SYSTEM:
            new_messages.insert(0, Message(role=SYSTEM, content=self.system_message))
        else:
            if isinstance(new_messages[0][CONTENT], str):
                new_messages[0][CONTENT] = self.system_message + '\n\n' + new_messages[0][CONTENT]
            else:
                # 已有 system 但 multimodal：拼到内容前部
                new_messages[0][CONTENT] = [ContentItem(text=self.system_message + '\n\n')] + new_messages[0][CONTENT]

    for rsp in self._run(messages=new_messages, **kwargs):
        for i in range(len(rsp)):
            if not rsp[i].name and self.name:
                rsp[i].name = self.name
        if _return_message_type == 'message':
            yield [Message(**x) if isinstance(x, dict) else x for x in rsp]
        else:
            yield [x.model_dump() if not isinstance(x, dict) else x for x in rsp]

# agent.py:133-148  _run 强制子类实现
@abstractmethod
def _run(self, messages: List[Message], lang: str = 'en', **kwargs) -> Iterator[List[Message]]:
    raise NotImplementedError

# agent.py:263-269  最简子类
class BasicAgent(Agent):
    def _run(self, messages, lang='en', **kwargs):
        extra_generate_cfg = {'lang': lang}
        if kwargs.get('seed') is not None:
            extra_generate_cfg['seed'] = kwargs['seed']
        return self._call_llm(messages, extra_generate_cfg=extra_generate_cfg)
```

**为什么不那样**：另一选择是把 `run` 合并进 `_run`，让子类自己处理类型归一。本系统拒绝——子类的每次重写都要重复 deepcopy / 类型判定 / lang 自动推断 / system 注入 / yield 协议转换。把这类"对所有 Agent 都一致的事"放基类，让子类只关心业务流。

**踩坑清单**：
- `run` 内 `messages = copy.deepcopy(messages)` —— caller 传入的 messages 被深拷贝，**子类 `_run` 内对 messages 的修改不会回传 caller**。这是好事，但若你刻意想"在 Agent 内部改 caller 的 messages"做不到，要在 caller 侧做。
- `Agent` 没有 `__init__` 后的字段约束——子类可在 `_run` 内动态加属性。但 `Assistant` 等构造时必须传 `llm`，传 `None` 会到 `_call_llm` 时才 raise `AttributeError: 'NoneType' object has no attribute 'chat'`——**延迟报错**，caller 调试时容易找不到病灶。
- `_run` 输出的 message 必须是 `List[Message]`，单条也要包成 list。`yield Message(...)` 直接抛会在 `run` 的 `for rsp in self._run(...)` 时因 `for i in range(len(rsp))` 报错——TypeError 不是文档级契约，要看源码。
- `BasicAgent` 是最简子类，但实际不会直接拿来用——它没工具、没 RAG，只是 demo。生产常用 `Assistant`（§8 ReAct 对照）。
- 子类要小心 system message 注入：若 `_run` 内部又 `insert(0, Message(SYSTEM, ...))`，会被 `run` 已经注入的那条消息覆盖成第二条 system——多 system 行为 LLM 解析时通常只看第一条（不同后端行为差异）。**规范做法是只给 `Agent.__init__` 传 `system_message`，`_run` 内不再加**。
- `run` 的 `lang` 推断在 messages 全英文但预期中文回复时也会走 `lang='en'`。要在 caller kwargs 里显式 `lang='zh'`。

## §3. 工具注册：三种入口对应三类来源

**决策点**：`Agent._init_tool` 是工具落注册的唯一入口，三分支覆盖三种工具来源：(a) 已构造的 BaseTool 实例 (b) MCP server 描述 dict (c) 注册表字符串名。

**代码证据**（`agent.py:212-237`）：
```python
def _init_tool(self, tool: Union[str, Dict, BaseTool]):
    if isinstance(tool, BaseTool):
        tool_name = tool.name
        if tool_name in self.function_map:
            logger.warning(f'Repeatedly adding tool {tool_name}, will use the newest tool in function list')
        self.function_map[tool_name] = tool                  # 入口 a：直接挂 BaseTool 实例

    elif isinstance(tool, dict) and 'mcpServers' in tool:
        tools = MCPManager().initConfig(tool)                # 入口 b：MCP 描述
        for tool in tools:
            ...
            self.function_map[tool_name] = tool

    else:
        if isinstance(tool, dict):
            tool_name = tool['name']
            tool_cfg = tool
        else:
            tool_name = tool                                  # 入口 c：纯字符串名
            tool_cfg = None
        if tool_name not in TOOL_REGISTRY:
            raise ValueError(f'Tool {tool_name} is not registered.')

        if tool_name in self.function_map:
            logger.warning(f'Repeatedly adding tool {tool_name}, will use the newest tool in function list')
        self.function_map[tool_name] = TOOL_REGISTRY[tool_name](tool_cfg)
```

**为什么不那样**：另一选择是统一成"传 dict 描述"——但 BaseTool 实例已构造好的情况（如自定义带状态的 Tool）必须允许直接挂入；名字字符串这条覆盖了" массовое使用 registry" 的简洁路径。三条分支是面向三种实际场景的最小完备集。

**踩坑清单**：
- 重复 add 同名 tool **不报错**只 warn，且会用 function_list 中**最后一个**覆盖前者（行 223、236）。若你希望"同名 tool 注册即冲突"，要自己写检查。
- `_init_tool` 的 `isinstance(tool, dict) and 'mcpServers' in tool` 这条优先于纯 dict cfg——**自定义 Tool 的 cfg dict 里恰好有 'mcpServers' key 会进 MCP 分支报错**，要选别的字段名。
- 注册的 Tool 名称必须与 `TOOL_REGISTRY` key 完全一致（大小写敏感），且 Tool 类必须 `@register_tool('name')` 装饰。**装饰器漏加**会 silent：Tool 类没在 registry，`_init_tool` 走到 `if tool_name not in TOOL_REGISTRY` raise `ValueError`——这条会立刻显式报错，比抽象类系统更早暴露问题，反而友好。
- `Agent.function_map` 是实例 attribute——同名 Agent 不同实例可以挂同名不同 cfg 的 Tool。但 cross-agent 注入 Tool 实例时 Care：Tool 自己的内部状态（如 DocParser 缓存的 Storage）是 Tool 实例 attribute，不是 Agent attribute——多 Agent 共用同一个 Tool 实例时 cache 共用，确认这是你想要的。

## §4. LLM 接入：`dict → BaseChatModel` 工厂路由

**决策点**：Agent 构造时 `llm` 既可以是 dict 也可以是已构造的 `BaseChatModel` 实例；dict 触发 `get_chat_model(llm)` 工厂路由到具体子类。

**代码证据**（`agent.py:38-59`、`llm/base.py:32`、`llm/__init__.py`）：
```python
# agent.py:56-59
if isinstance(llm, dict):
    self.llm = get_chat_model(llm)
else:
    self.llm = llm

# llm/base.py:32
LLM_REGISTRY = {}

# llm/__init__.py（工厂）
def get_chat_model(cfg):
    model_type = cfg.get('model_type', 'qwen_dashscope')
    return LLM_REGISTRY[model_type](cfg)
```

**为什么不那样**：另一选择是 Agent 强制要求传 `BaseChatModel` 实例——caller 必须自己 `get_chat_model`。本系统让 Agent 一并兼容 dict 是为了 examples 简洁（`Assistant(llm={'model': 'qwen-max'})` 一行起），代价是 Agent 多了一次 isinstance 判断。

**踩坑清单**：
- `get_chat_model(dict)` 默认 `model_type='qwen_dashscope'`——若你传 OpenAI 兼容后端必须显式 `model_type='openai'` 或 `model_type='oai'`，否则默认走 dashscope 报 token 错。
- `BaseChatModel` 的抽象方法：`chat()` 公开接口 + `_preprocess_messages` / `_chat_with_functions` / `_postprocess_messages` 三个 protected 方法（详见 07 章）。新模型后端要全部实现。
- `LLM_REGISTRY` 是全局 dict，被 `@register_llm('name')` 装饰器写入。**多个 Python 进程**不会共享 registry——如 pytest 多 process 跑测试时各 worker 自己注册。生产单进程时无此问题。
- llm 配置 dict 里能不能塞 secret？可以——`api_key` 是常见字段，但日志和 traceback 里如果 Tool 报错会打印 dict。生产上建议从 env 读后塞，不在 yaml 配置文件持久化 api_key。
- Agent 把 cfg 实例化后只存 `self.llm`，不存原始 cfg——**Agent pickle 或 deep copy 时不会重建 LLM 实例**，会把 BaseChatModel 实例一起拷贝。

## §5. 异常处理：`_call_tool` 把 Exception 转字符串回喂 LLM

**决策点**：Agent 调用 Tool 时，若 Tool 抛 `ToolServiceError` 或 `DocParserError` **透传 raise** 给 caller；其他 Exception 则**捕获转为字符串 error_message 回喂 LLM**。

**代码证据**（`agent.py:178-210`）：
```python
def _call_tool(self, tool_name, tool_args='{}', **kwargs):
    if tool_name not in self.function_map:
        return f'Tool {tool_name} does not exists.'
    tool = self.function_map[tool_name]
    try:
        tool_result = tool.call(tool_args, **kwargs)
    except (ToolServiceError, DocParserError) as ex:
        raise ex                                              # ★ 服务级错误透传
    except Exception as ex:
        exception_type = type(ex).__name__
        exception_message = str(ex)
        traceback_info = ''.join(traceback.format_tb(ex.__traceback__))
        error_message = f'An error occurred when calling tool `{tool_name}`:\n' \
                        f'{exception_type}: {exception_message}\n' \
                        f'Traceback:\n{traceback_info}'
        logger.warning(error_message)
        return error_message                                   # ★ 一般异常转字符串回喂 LLM

    if isinstance(tool_result, str):
        return tool_result
    elif isinstance(tool_result, list) and all(isinstance(item, ContentItem) for item in tool_result):
        return tool_result                                     # 多模态结果原样返回
    else:
        return json.dumps(tool_result, ensure_ascii=False, indent=4)
```

**为什么不那样**：另一选择是"所有 Tool 异常都透传 raise"。但 LLM Agent 的常见情况是 Tool 偶发出错（如 code_interpreter 沙箱超时、网络抖动），让 LLM 看到 "Tool 调用失败" 的字符串后能**自行决策**——重试、换参数、或告诉用户失败。透传 raise 会让整个对话 crash。`ToolServiceError`/`DocParserError` 是"基础设施级"故障，**透传够 LLM 看也没用**——MCP server 都崩了 LLM 修不了。

**踩坑清单**：
- **exception 的 traceback 完整被打回 LLM** —— 如果 Tool 实现里有敏感信息（如密钥、内部路径），可能被泄漏到 LLM 上下文。生产副作用：tool 实现要 redact 自己的 exception。
- 这是**靠 LLM 自愈**的设计——但不是所有 LLM 都能把 "Tool X 报错" 解读为"换一种调用方式"。Qwen3-Coder 这类推理模型可以，老一代 Qwen2.5-Instruct 时不时会卡在循环里反复用同一参数重试。
- logger.warning 但不 raise 等于**静默吞**——caller 看不到 Tool 出错，要监控 APM 才知道。运维侧要订阅 warning 日志。
- 自定义 Tool 要选好继承的异常类：希望"LLM 可见、可重试"就 raise 普通 Exception；希望"立即终止整个 Agent 流"就 raise `ToolServiceError`。
- `_detect_tool` 不在这里做异常分类——它只判断 LLM 输出是否含 function_call（`agent.py:239-259`），任何异常处理都由 `_call_tool` 接管。

## §6. 全局注册表：`ToolRegistry / LLMRegistry / MCPManager` 三套并存

**决策点**：Qwen-Agent 用三套全局注册表覆盖三类对象：

| 注册表        | 位置                  | 装饰器            | 用途                       |
|---------------|-----------------------|-------------------|----------------------------|
| `TOOL_REGISTRY`   | `tools/base.py:24`    | `@register_tool`     | 内置 + 自定义 Tool 类      |
| `LLM_REGISTRY`    | `llm/base.py:32`      | `@register_llm`     | 模型后端                    |
| `MCPManager`      | `tools/mcp_manager.py` | 单例 + `initConfig` | 动态 MCP server 工具        |

**代码证据**：
```python
# tools/base.py:24
TOOL_REGISTRY = {}

# tools/base.py:28 左右
def register_tool(name):
    def decorator(cls):
        TOOL_REGISTRY[name] = cls
        return cls
    return decorator

# tools/mcp_manager.py (iterinit 单例)
class MCPManager:
    _instance = None
    def __new__(cls, *args, **kwargs):
        if cls._instance is None:
            cls._instance = super().__new__(cls, *args, **kwargs)
        return cls._instance

    def initConfig(self, mcp_servers_config) -> List[BaseTool]:
        ...   # 启动子进程 / stdio 连接 MCP server，返回 Tool 实例
```

**为什么不那样**：另一选择是用"插件入口点"（entry_points / setuptools plugins）做组件加载。Qwen-Agent 不选——框架定位于小而专，要求所有 Tool 与 LLM 在 import 时即可被发现。注册表的代价是**全局可变状态**，对测试不友好。

**踩坑清单**：
- 三套全是**进程级全局**——pytest 做 fixture 隔离时要主动清空，否则不同测试互相污染。生产单进程应用无此问题。
- `MCPManager` 单例 + 跨 Agent 共享：两个 Agent 实例共用同一个 MCPManager，**MCP server 子进程生命期与 Python 进程一致**，Agent 反复构造销毁不会 cleanup MCP 连接。长跑实例要警觉资源泄漏。
- `MCPManager().initConfig(tool)` 的 dict 形如 `{'mcpServers': {...}}`——这与 OpenAI / Anthropic 推的 MCP client config 格式兼容，但 dict 里若 `servers` 数量大，启动多个子进程慢。
- 注册表 lookup 是 O(1) dict 取，但有**拼写敏感**——名字错会 silent 直到你 `_init_tool` 走到 `if tool_name not in TOOL_REGISTRY: raise`。

## §7. 何时用 MCP、何时用内置注册 Tool

**决策点**：内置 Tool 走 `@register_tool('name')`，框架 import 时即注册；MCP tool 走 `Agent(function_list=[{'mcpServers': {...}}])`，运行时按需启动 MCP server 进程拉 Tool 列表。

**为什么选哪个**：

| 维度        | 内置 Tool             | MCP Tool                       |
|-------------|-----------------------|--------------------------------|
| 启动成本    | 0（进程内调用）       | 子进程冷启动 + stdio 握手      |
| 状态共享    | Tool 实例属性，跨调用持久 | MCP server 进程独立，状态留在子进程 |
| 跨语言      | Python only           | MCP server 可任何语言           |
| 跨进程隔离  | 无（Agent 崩 Tool 崩）  | 有（MCP server 崩 Agent 不崩）  |
| Tool 数量    | 几个到几十             | 几十到几百（生态）             |
| 重用生态    | 自定义实现             | 接入社区现有 MCP server         |

**踩坑清单**：
- 调用一次 MCP Tool 会过 IPC（stdio 或 SSE），**单次 call 延迟比内置 Tool 高**——约 20~50ms IPC + server 执行时间。FnCall 循环里反复调 MCP Tool 会显著掉性能。
- MCP server 出错抛 `ToolServiceError` —— 透传 raise（§5），**Agent 流会中断**。这合理但要预估：如果 MCP server 是第三方不可控的，每次 Agent run 都要 try/except 包一层。
- 内置 Tool 的 `cfg` 字段在 `@register_tool` 后 `_init_tool` 取 cfg dict 用，但 MCP tool 的 cfg 在 MCPManager 层处理——**MCP tool 不能像内置 Tool 那样初始化时注入 cfg**。
- 开发期：先用内置 Tool 跑通原型，再考虑是否做 MCP 化。**多数 Tool 不值得 MCP 化**——MCP 的核心价值是"复用别人写的 server"。

## §8. FnCall vs ReAct：两条工作流为何都保留

**决策点**：Qwen-Agent 同时保留 `FnCallAgent`（Function Call 模型原生范式）和 `ReActChat`（Thought→Action→Observation 文本循环）。前者依赖模型支持 FnCall，后者纯文本 prompt 也能跑。

**代码证据**：
```python
# agents/fncall_agent.py:27-100（核心循环）
class FnCallAgent(Agent):
    def _run(self, messages, lang='en', **kwargs):
        ...
        while True:
            ...
            use_tool, tool_name, tool_args, _ = self._detect_tool(out)   # 100 行左右
            if use_tool:
                tool_result = self._call_tool(tool_name, tool_args, messages=messages, **kwargs)
                ...   # 把 tool result 拼 messages 继续循环
            else:
                yield ...   # 没工具调用 → 输出 return
                return

# agents/react_chat.py:50-100
class ReActChat(FnCallAgent):
    PROMPT_REACT = ...                              # ReAct 文本模板
    def _run(self, messages, lang='en', **kwargs):
        ...   # 用文本 Thought/Action/Observation 而非 function_call 字段
```

**为什么不那样**：另一选择是只保留 FnCall——但**多数开源小模型不支持原生 FnCall**，Qwen-Agent 要兼容 OpenAI FnCall API 之外的 ecosystem，就需要 ReAct 这条文本路径。两条并存是兼容性而非冗余。

**踩坑清单**：
- 用 `FnCallAgent` 时模型必须支持 function_call 字段输出。**Qwen2.5-Instruct 默认不支持原生 FnCall**，要么用 nous 格式 prompt 模板（03 章），要么退回 ReAct。Qwen3 系列已原生支持。
- ReAct 的 prompt 教模型按 `Thought:\nAction:\nAction Input:\nObservation:\n` 文本格式输出——**任何模型都能跑但更易失败**：parse 错了 Thought 就卡循环。`react_chat.py` 末尾有兜底正则但 tolerance 取决于模型稳定性。
- FnCall 循环是**无限循环**——`while True`，依赖 LLM 主动 yield（即输出不含 function_call 才 return）。如果 LLM 反复调同一 Tool 同一参数，会死循环——`fncall_agent.py` 里有 `max_turn` 之类约束（具体值看版本），但默认较高，长循环会耗 LLM 配额。
- ReAct 子类继承 FnCallAgent 而非直接 Agent——`agents/react_chat.py:50` `class ReActChat(FnCallAgent)`。这条继承表明 ReAct 复用 FnCall 的工具调用骨架只重写 `_run` 与 prompt。但 `_detect_tool` 在 FnCall 路径里靠 message.function_call 字段，ReAct 是文本解析——**这意味着 ReActChat 调 `self._detect_tool` 会一直返回 False**（因为没 function_call 字段），ReActChat 必须自己 override 这个判断。

## §10. 改这块代码前的检查清单

如果你要扩展或修改 Agent 基类、工具注册、模型接入，逐项确认：

- [ ] 新 Agent 子类：实现 `_run` 返回 `Iterator[List[Message]]`，单条也包成 list
- [ ] 新 Agent 子类：构造时把 `llm` 当必传，传 None 会在 `_call_llm` 时延迟报错
- [ ] 新 Agent 子类：`_run` 内不要重复注入 system message——`run` 已经注入
- [ ] 新 Tool：用 `@register_tool('name')` 装饰，`name` 严格匹配类 attribute `name`
- [ ] 新 Tool 的 cfg 字段名不能叫 `mcpServers`——会被 `_init_tool` 走到 MCP 分支
- [ ] 新 Tool 自定义异常：基类继承 `Exception` 会被吞回喂 LLM；继承 `ToolServiceError` 会透传 raise 中断 Agent
- [ ] 新模型后端：实现 `chat + _preprocess + _chat_with_functions + _postprocess` 四接口；用 `@register_llm('your_model')` 注册
- [ ] 切换 MCP Tool：评估跨进程延迟，对 FnCall 循环热点别用
- [ ] 切换 ReAct 模式：确认模型不支持原生 FnCall，否则优先用 FnCall 路径效率更高
- [ ] 测试注册表隔离：pytest 多 process 跑时各 worker 自己注册，不共享 LLM_REGISTRY
